## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps (`gap-1`), inline padding |
| sm | 8px | Compact element spacing (`gap-2`, `py-2`) |
| md | 16px | Default element spacing (`gap-4`, `px-4`) |
| lg | 24px | Section padding (`px-6`, `py-6`) |
| xl | 32px | Layout gaps |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

Phase 2 introduces NO new spacing values. All new UI elements use the 8-point scale above. Existing codebase elements that use non-standard spacing (e.g. `px-5`, `px-3`) are inherited and not redeclared here.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body / Label | 11px (`text-[11px]`) | 500 (medium) | 1.5 (default) |
| Button text / Badge | 10px (`text-[10px]`) | 700 (bold) | 1.5 |
| Context menu item | 12px (`text-[12px]`) | 500 (medium) | 1.5 |

Phase 2 introduces NO new type sizes. Two weights only: 500 (medium) for body, labels, and context menu items; 700 (bold) for headings, buttons, and badges.

## Visual Focal Point

Primary visual anchor: the role-conditional action button (Assign / Claim Thread / Reassign) is the focal point of the action bar. The read-only branch (member viewing another user's thread) intentionally removes this anchor, leaving the assignee display as a passive indicator with no interactive focal point.

## Component Inventory

Phase 2 modifies existing components. No new standalone components are created.

### Modified: `_thread_detail.html` Action Bar

**Current state:** Two branches -- `{% if is_admin %}` shows assign dropdown, `{% else %}` shows claim/unassigned badge.

**Phase 2 target:** Four branches based on role and thread ownership.

| Branch | Condition | UI |
|--------|-----------|-----|
| 1. Admin/Gatekeeper | `can_assign` is True | Full "Assign to" dropdown with all active users + optional note + "Assign" button (existing, unchanged except variable rename) |
| 2. Member, own thread | `not can_assign` AND `thread.assigned_to == request.user` | Assignee display (avatar + name) + "Reassign" button that reveals inline form |
| 3. Member, unassigned, has visibility | `not can_assign` AND `can_claim` is True | "Claim Thread" button (existing, unchanged) |
| 4. Member, unassigned, no visibility | `not can_assign` AND `claim_disabled` is True | Disabled "Claim Thread" button with `title` tooltip |
| 5. Member, others' thread | `not can_assign` AND `thread.assigned_to` AND `thread.assigned_to != request.user` | Read-only assignee display (avatar + name), no action buttons |

## Interaction Contracts

### Reassign Flow (Member on Own Thread)

1. Member sees their assigned thread detail panel with assignee display + "Reassign" button
2. Member clicks "Reassign" -- JavaScript toggles `display: none` to `display: block` on the reassign form div
3. Form shows: select dropdown (category-filtered users) + reason textarea + "Reassign Thread" / "Keep Thread" buttons
4. Member selects assignee, types reason, clicks "Reassign Thread"
5. HTMX POSTs to `/emails/threads/<pk>/reassign/` with `assignee_id` + `reason`
6. Server validates: ownership, non-empty reason, assignee has CategoryVisibility
7. On success: returns updated detail panel HTML + OOB thread card HTML (standard pattern)
8. On failure (403): HTMX swap shows error toast in detail panel header area (same pattern as existing `toast_msg`)
9. "Keep Thread" button: JavaScript hides the form div, no server request

### Claim Flow (Unchanged)

Existing behavior preserved. New addition: disabled state when thread category is outside member's visibility.

1. Disabled button: `<button disabled title="This thread is outside your assigned categories" class="... opacity-50 cursor-not-allowed">`
2. Tooltip implementation: native `title` attribute (no custom tooltip library -- consistent with existing `title` tooltips on badges)

### Context Menu Reassign

1. Member right-clicks own thread card -- context menu shows "Reassign..." item
2. Clicking "Reassign..." opens the detail panel (same as "Assign to..." does for admins -- `hx-get` to detail endpoint)
3. Detail panel loads with reassign form visible (or member clicks "Reassign" button to reveal it)
4. Keyboard shortcut: R key triggers the same action when context menu is open

### Read-Only View (Member on Others' Thread)

1. Member opens a thread assigned to someone else
2. Detail panel shows: subject, badges (priority/status/category as plain text, not editable dropdowns), assignee display (read-only), SLA bar, messages, activity log, notes
3. Member CAN: read all content, view activity log, add internal notes
4. Member CANNOT: change assignment, change status, change priority, change category
5. No action buttons in the action bar section (acknowledge, close, assign, claim, reassign all hidden)
